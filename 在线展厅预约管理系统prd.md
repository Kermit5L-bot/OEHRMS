## 2. 项目背景

公司目前拥有多个线下展厅及实训基地展厅，包括北京公司展厅、西安公司展厅、阜阳实训基地展厅、新疆实训基地展厅。客户在参观前通常需要通过销售、市场部或其他人员进行沟通预约，预约信息容易分散在微信、电话、表格或人工记录中，存在流程不统一、审批不清晰、跟进不及时、数据沉淀不足等问题。

本项目希望建设一个轻量级的在线展厅预约管理系统，让客户能够在线了解展厅信息、提交参观预约和联系方式，市场部可在后台统一查看、审批、备注和跟进预约，从而提升客户接待流程效率和服务体验。

---

## 3. 项目目标

### 3.1 核心目标

第一版系统的核心目标是：跑通“客户在线预约 → 市场部后台审批 → 市场部跟进接待”的完整流程。

### 3.2 业务目标

1. 建立统一的展厅预约入口。
2. 降低市场部人工收集预约信息的工作量。
3. 统一管理北京、西安、阜阳、新疆四个展厅的预约记录。
4. 提升客户预约体验，减少反复沟通成本。
5. 沉淀客户留资数据，便于后续市场跟进。
6. 让市场部可以清晰掌握预约状态、审批结果和接待备注。

### 3.3 第一版目标

第一版不追求复杂功能，重点实现以下闭环：

- 用户可以查看展厅信息。
- 用户可以选择展厅并提交预约。
- 用户可以提交姓名、公司、联系方式、参观时间、参观人数等留资信息。
- 后台人员可以登录系统。
- 后台人员可以查看预约列表。
- 后台人员可以审批预约。
- 后台人员可以维护接待备注。
- 后台人员可以查看和管理客户留资信息。

## 5. 用户角色

### 5.1 访客 / 客户

客户无需登录即可访问用户端页面。

客户可完成：

- 浏览首页。
- 查看展厅列表。
- 查看展厅详情。
- 选择展厅并提交预约。
- 填写个人及公司信息。
- 查看提交成功提示。

### 5.2 市场部人员

市场部人员需要登录后台。

市场部人员可完成：

- 查看全部预约记录。
- 按展厅、状态、时间筛选预约。
- 查看预约详情。
- 审批预约，包括通过、拒绝。
- 填写审批意见。
- 填写接待备注。
- 查看客户留资信息。
- 维护展厅基础信息。

### 5.3 系统管理员

第一版可先将系统管理员和市场部人员合并处理。若需要区分，则系统管理员拥有全部权限。

系统管理员可完成：

- 管理后台账号。
- 管理展厅信息。
- 管理所有预约记录。

第一版建议只内置 1 个管理员账号，降低开发复杂度。

## 6. 核心业务流程

### 6.1 客户预约流程

1. 客户进入系统首页。
2. 客户查看展厅介绍和展厅列表。
3. 客户选择某个展厅，进入展厅详情页。
4. 客户点击“预约参观”按钮。
5. 系统进入在线预约页，并自动带入当前展厅信息。
6. 客户填写预约信息。
7. 客户提交预约。
8. 系统校验表单信息。
9. 校验通过后，系统生成一条预约记录，状态为“待审批”。
10. 客户进入提交成功页。
11. 提交成功页提示“预约已提交，工作人员将尽快与您联系”。

### 6.2 市场部审批流程

1. 市场部人员登录后台。
2. 进入预约管理页面。
3. 查看待审批预约。
4. 点击预约记录查看详情。
5. 根据客户信息、预约时间、展厅安排进行审批。
6. 审批结果为“通过”或“拒绝”。
7. 若通过，可填写接待安排和备注。
8. 若拒绝，需填写拒绝原因。
9. 系统更新预约状态。
10. 市场部可后续补充接待备注。

### 6.3 留资沉淀流程

1. 客户提交预约表单。
2. 系统自动保存客户姓名、公司、手机号、职位、需求描述等信息。
3. 后台留资管理中展示客户数据。
4. 市场部可按展厅、提交时间、客户公司、手机号进行查询。
5. 留资记录与预约记录关联。

## 7.4 在线预约页

### 页面目标

让客户提交完整预约和留资信息。

### 表单字段

| 字段 | 类型 | 是否必填 | 说明 |
|---|---|---|---|
| 预约展厅 | 下拉选择 | 是 | 默认可从详情页带入 |
| 参观日期 | 日期选择 | 是 | 不允许选择过去日期 |
| 参观时间段 | 下拉选择 | 是 | 上午 / 下午，可后续扩展 |
| 参观人数 | 数字输入 | 是 | 最小 1 人 |
| 客户姓名 | 文本输入 | 是 | 预约联系人姓名 |
| 手机号码 | 文本输入 | 是 | 需进行手机号格式校验 |
| 公司名称 | 文本输入 | 是 | 客户所在公司 |
| 职务 | 文本输入 | 否 | 客户职位 |
| 所属行业 | 文本输入 / 下拉选择 | 否 | 可选字段 |
| 参观目的 | 多行文本 | 否 | 客户需求说明 |
| 是否需要接待讲解 | 单选 | 否 | 默认是 |
| 备注 | 多行文本 | 否 | 其他补充说明 |

### 表单校验规则

1. 预约展厅不能为空。
2. 参观日期不能为空，且不能早于当前日期。
3. 参观时间段不能为空。
4. 参观人数不能为空，且必须大于 0。
5. 客户姓名不能为空。
6. 手机号码不能为空，需符合手机号格式。
7. 公司名称不能为空。
8. 文本字段需要去除前后空格。
9. 防止重复点击提交按钮。

### 提交逻辑

客户点击“提交预约”后：

1. 前端进行表单校验。
2. 校验失败时，在对应字段下方显示错误提示。
3. 校验通过后，请求后端创建预约。
4. 后端创建预约记录。
5. 预约初始状态为“待审批”。
6. 创建成功后跳转提交成功页。

## 8.5 审批功能

### 审批通过

市场部点击“通过预约”后，弹出审批弹窗。

字段：

| 字段 | 类型 | 是否必填 |
|---|---|---|
| 审批意见 | 多行文本 | 否 |
| 接待负责人 | 文本输入 | 否 |
| 接待备注 | 多行文本 | 否 |

提交后：

- 预约状态变为“已通过”。
- 记录审批人。
- 记录审批时间。
- 保存审批意见。
- 保存接待备注。

### 审批拒绝

市场部点击“拒绝预约”后，弹出拒绝弹窗。

字段：

| 字段 | 类型 | 是否必填 |
|---|---|---|
| 拒绝原因 | 多行文本 | 是 |

提交后：

- 预约状态变为“已拒绝”。
- 记录审批人。
- 记录审批时间。
- 保存拒绝原因。

### 状态限制

| 当前状态 | 可操作 |
|---|---|
| 待审批 | 通过、拒绝、编辑备注 |
| 已通过 | 标记完成、编辑备注 |
| 已拒绝 | 查看、编辑备注 |
| 已完成 | 查看、编辑备注 |
| 已取消 | 查看 |

### 数据来源

### 列表字段

| 字段 | 说明 |
|---|---|
| 客户姓名 | 预约联系人 |
| 手机号 | 联系电话 |
| 公司名称 | 客户公司 |
| 职务 | 客户职位 |
| 所属行业 | 客户行业 |
| 最近预约展厅 | 最近一次预约的展厅 |
| 最近预约时间 | 最近提交时间 |
| 预约次数 | 同手机号或同公司累计预约次数 |
| 跟进备注 | 市场部维护 |

## 14. 非功能需求

## 14.1 响应式适配

用户端需支持：

- 桌面浏览器
- 手机浏览器
- 平板浏览器

后台优先适配电脑端，手机端可基本可用，不作为第一版重点。

## 14.2 性能要求

- 首页首屏加载应尽量控制在 3 秒内。
- 预约提交接口响应建议控制在 1 秒内。
- 列表查询支持分页，避免一次性加载过多数据。

## 14.3 安全要求

- 后台必须登录后访问。
- 密码不得明文存储，应使用哈希加密。
- 表单输入需做基础 XSS 防护。
- 后台接口需校验登录态。
- 手机号等客户信息仅后台可见。

## 14.4 数据要求

- 预约记录提交后不可丢失。
- 审批操作需记录审批人和审批时间。
- 重要时间字段统一使用服务器时间。
- 删除功能第一版尽量不做，避免误删数据。

## 14.5 易用性要求

- 表单错误提示清晰。
- 提交按钮需有 loading 状态。
- 后台列表需支持按状态快速筛选。
- 待审批状态需要明显突出。

## 17. 推荐技术实现方案

考虑到项目由个人借助 AI 编程工具完成，第一版建议选择简单、成熟、开发效率高的方案。

### 17.1 推荐方案 A：Next.js 全栈方案

适合希望前后端一体化、快速上线的情况。

技术栈：

- 前端：Next.js + React + Tailwind CSS
- 后台 UI：shadcn/ui 或 Ant Design
- 后端：Next.js API Routes / Server Actions
- 数据库：PostgreSQL 或 SQLite
- ORM：Prisma
- 部署：Vercel / Railway / Render / 自有服务器

优点：

- 前后端一套项目。
- AI 编程工具支持度高。
- 页面和接口可以统一管理。
- 适合 MVP 快速开发。

### 17.2 推荐方案 B：Vue + Node.js 方案

适合熟悉 Vue 技术栈的情况。

技术栈：

- 前端：Vue 3 + Vite + Element Plus
- 后端：Node.js + Express / NestJS
- 数据库：MySQL / PostgreSQL
- ORM：Prisma

优点：

- 后台管理系统开发体验成熟。
- Element Plus 适合快速搭建管理后台。

### 17.3 最低成本建议

如果目标是“最低成本、先可用”，建议：

- Next.js + Tailwind CSS + Prisma + SQLite/PostgreSQL。
- 第一版使用账号密码登录。
- 第一版预置 4 个展厅。
- 第一版不做短信、微信、企业微信通知。
- 第一版不做复杂权限。
- 第一版不做图片上传，先使用图片 URL 或静态图片。

## 20. 风险与注意事项

### 20.1 业务风险

1. 预约提交后如果没有通知机制，市场部可能不能及时发现。
   - 第一版解决方案：后台首页突出待审批数量。
   - 后续优化：增加企业微信或邮件通知。

2. 多个客户选择同一时间段可能冲突。
   - 第一版解决方案：市场部人工审批。
   - 后续优化：增加时间段容量限制。

3. 客户手机号涉及隐私。
   - 第一版解决方案：后台登录可见，用户端成功页脱敏展示。

### 20.2 开发风险

1. 功能一次性做太多，导致开发周期拉长。
   - 建议严格控制第一版范围。

2. 过早做复杂权限、通知、导出、图片上传，会增加成本。
   - 建议第一版先跑通流程。

3. AI 编程工具容易遗漏边界校验。
   - 建议每个阶段开发后都按验收标准测试。

1. 客户可以通过电脑或手机访问系统。
2. 客户可以查看 4 个展厅。
3. 客户可以提交预约。
4. 后台可以看到客户提交的预约。
5. 市场部可以审批通过或拒绝预约。
6. 市场部可以填写接待备注。
7. 客户留资可以在后台沉淀和查询。
8. 展厅基础信息可以在后台维护。
9. 系统数据能够正常保存，不因刷新页面丢失。
10. 核心流程可以连续演示：查看展厅 → 提交预约 → 后台审批 → 添加接待备注。

## 22. 建议的第一版开发优先级

| 优先级 | 模块 | 说明 |
|---|---|---|

| P1 | 展厅管理 | 支撑内容维护 |
| P2 | 工作台统计 | 提升后台效率 |
| P2 | 导出、通知、日志 | 后续增强 |
